# Готовность к запуску (сводка аудита + репозиторий)

Документ связывает **внешнюю оценку продукта** с правилами **`AGENTS.md`**, чеклистом **`docs/TODO_PRODUCTION.md`** и архитектурой.

## Продукт и scope

| Вердикт | Действие |
|--------|----------|
| Сильная ценность для UK self-employed; риск расползания scope | **MVP-фокус:** MTD + транзакции + инвойсы + банк-синк по кнопке. Остальное — **фичефлаги / позже** (mortgage, voice, много языков до подтверждения PMF). |

## Устранённые противоречия в документации

| Было | Стало |
|------|--------|
| В `README` mtd-agent назван «auto-submission» | Формулировка **prepare → user confirms → submit**, без «автосабмита» |
| В `AGENTS.md` таблица сервисов: «auto-submission agent» | Описание workflow + явная отсылка к правилу **explicit confirmation** |

Правило неизменно: **`AGENTS.md` — HMRC submit только после явного подтверждения пользователя.**

## Архитектура (33 сервиса)

- **Плюс:** модульность, параллельная разработка.
- **Минус:** стоимость эксплуатации. Практика: **production slice** — минимальный набор сервисов для релиза; остальное не в основном compose или за profile (см. `graphql-gateway` с `--profile graphql`).

## Must-have перед публичным production (кратко)

1. **Технический запрет на «тихие» действия:** bank sync только по действию пользователя; HMRC — **двухфазно** (prepare → confirm → submit) с **audit** согласия на бэкенде (`docs/TODO_PRODUCTION.md` §9, §13).
2. **Миграции Alembic:** не полагаться только на «upgrade при старте контейнера» в проде — отдельный job/init (известный gotcha в `AGENTS.md`).
3. **Наблюдаемость:** 5xx, p95, ошибки БД, глубина очередей; correlation-id через gateway.
4. **Scope релиза:** mortgage / AI voice / любые формулировки «авто» для HMRC — скрыть или пометить beta до стабилизации.
5. **E2E smoke:** регистрация → банк → sync → транзакции → подготовка MTD → **confirm** → submit (sandbox).
